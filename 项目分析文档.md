# robust-waste-detection-main 项目阅读与分析

## 1. 项目定位

该项目对应论文 **Robust and Label-Efficient Deep Waste Detection (BMVC 2025)**，核心目标是：

- 在工业垃圾分拣场景（ZeroWaste 数据集）建立强基线；
- 提升开放词汇零样本检测（Zero-Shot OVOD）效果；
- 通过半监督伪标注利用无标注数据，超越纯监督训练性能。

从工程上看，它不是“单一模型训练脚本”，而是一个完整实验框架，覆盖：

1. 零样本评估（GroundingDINO / OWLv2 / YOLO-World）；
2. 全监督微调（基于 MMDetection + GroundingDINO 配置）；
3. 半监督伪标注生成与训练（多模型集成 + 共识加权）。

---

## 2. 项目整体结构（按功能）

基于仓库结构和 README，可归纳为以下模块：

- `zero_shot_ovod/`
  - 零样本评估模块；
  - 配置文件在 `configs/`；执行脚本在 `src/`；结果在 `evaluation_results/`。

- `semi_sup/`
  - 半监督相关模块；
  - 包含伪标注输入、融合脚本、工具函数；
  - 关键脚本：
    - `scripts/consolidate_pseudo_annotations.py`
    - `scripts/generate_ensemble_pseudo_annotations.py`
    - `utils/add_offset.py`（README 中用于训练前处理）

- `external_modules/`
  - 外部依赖源码（`mmdetection`、`GroundingDINO`）；
  - 主要承担训练、评估和模型实现；项目本身通过配置和脚本调用。

- `data/`
  - 期望数据目录（ZeroWaste-f 与 ZeroWaste-s）。

- `weights/`
  - 预训练和发布权重（通过 HuggingFace 下载）。

- `experiments/`
  - 训练/评估输出目录。

## 3. 技术路线总结

项目技术路线可以概括为“三阶段闭环”：

### 阶段 A：零样本 OVOD 基准

- 在 ZeroWaste-f 上评估 GroundingDINO / OWLv2 / YOLO-World；
- 对比“类名提示词（class-only）”与“优化提示词（optimized prompts）”；
- 目的：验证开放词汇模型在垃圾分拣场景中的可迁移性与提示词敏感性。

### 阶段 B：全监督强基线

- 使用 ZeroWaste-f 标注集对 GroundingDINO（Swin-T/Swin-B）微调；
- 基于 MMDetection 配置体系完成训练和评估；
- 形成稳定可复现实验基线（论文中约 51.6 mAP）。

### 阶段 C：半监督伪标注增强

- 使用多个模型在 ZeroWaste-s（无标注）上预测；
- 融合为高质量伪标签（WBF + IoU 聚类 + 模型共识加权）；
- 与有标注数据联合训练，达到超过全监督的效果（论文中最高约 54.3 mAP）。

## 4. 关键脚本深度解析（semi_sup）

你当前打开的两个脚本正是半监督流程核心。

### 4.1 `consolidate_pseudo_annotations.py`

作用：

1. 读取多个模型输出的 COCO 风格伪标注 JSON；
2. 给每条预测打上 `model_id`；
3. 统一格式并合并为一个“总伪标注文件”；
4. 输出统计（每模型预测数、过滤前后数量）。

特点：

- 输入文件固定为 4 个模型预测（co-detr / deta / dino / gdino）；
- 当前阈值 `THRESHOLD = 0.0`，默认不做置信度过滤；
- 输出到 `data/pseudo_labels/zerowaste-s_consolidated_pseudo_annotations.json`。

这一步是“伪标注标准化 + 汇总”，为下一步集成融合准备统一输入。

---

### 4.2 `generate_ensemble_pseudo_annotations.py`

作用：

在 consolidated 文件上执行“共识驱动的软伪标注融合”，输出最终可训练伪标签。

核心参数（脚本头部）：

- `tau = 0.05`：初筛置信度阈值；
- `theta = 0.65`：IoU 聚类阈值；
- `m = 2`：至少 2 个不同模型一致才保留；
- `tau_f = 0.35`：最终输出阈值；
- `alpha = 0.1`：空间分散惩罚；
- `beta = 0.05`：模型一致性奖励。

核心流程：

1. **低分过滤**：先去掉极低分候选；
2. **按图像+类别组织**：减少无关匹配；
3. **IoU 聚类**：将同一目标的跨模型框聚到一起；
4. **WBF 融合**：按分数加权融合 bbox；
5. **共识打分**：结合
   - 组内框与融合框的 IoU 离散度（spread）
   - 参与模型数（`num_models`）
   重新缩放分数；
6. **边界裁剪**：保证 bbox 不越界；
7. **最终阈值过滤**：保留高质量伪标签；
8. **导出 COCO JSON**：供半监督训练使用。

输出文件：

- `data/pseudo_labels/zerowaste-s_ensemble_consensus_pseudo_annotations.json`

这一步体现了项目的主要创新工程实现：不仅融合框坐标，还融合“跨模型一致性信号”。

---

### 4.3 `semi_sup/utils/consolidate_coco_annotations.py`

这是合并脚本的底层工具库，职责清晰：

- 读取 / 写入 JSON；
- 阈值过滤函数工厂；
- 合并时补齐 COCO 关键字段（`area`、`iscrowd`、类型规范等）；
- 输出过滤统计与预测分布统计。

优点：

- 逻辑模块化，后续可复用于其他数据集或模型组合。

## 5. 零样本模块（zero_shot_ovod）分析

从 `zero_shot_ovod/README.md` 看，设计重点在“可复现实验矩阵”：

- 模型隔离依赖（每模型独立环境）；
- 配置驱动（YAML 内嵌 prompts）；
- 输出标准化（`metrics.json` + `predictions.json`）。

其中 GroundingDINO / OWLv2 支持多套 prompt 集并自动评估，便于系统比较“提示词工程”收益。

## 6. 依赖与运行方式

根目录 `requirements.txt` 很简洁：

- `fairscale`
- `transformers`
- `tomli`（低版本 Python 兼容）

但项目真实依赖由两部分构成：

1. 根依赖（当前项目脚本）；
2. 外部模块依赖（MMDetection、GroundingDINO、各零样本模型环境）。

因此它本质是“多环境 + 多框架”项目，环境管理复杂度较高。

---

## 7. 工程优点

1. **实验链路完整**：从零样本到监督到半监督，一条线闭环；
2. **可复现性较强**：README 写清楚数据结构、命令、权重来源；
3. **模块划分合理**：`zero_shot_ovod` 与 `semi_sup` 关注点分离；
4. **伪标注质量控制较细**：引入 IoU spread 与模型共识，而非单纯阈值；
5. **兼容主流生态**：依托 MMDetection，配置和训练流程成熟。

## 8. 潜在问题与改进建议

### 8.1 目录与路径硬编码较多

如半监督脚本中输入文件列表固定，适合论文复现，不太适合通用化。

建议：

- 改为 CLI 参数（输入列表、阈值、输出路径）；
- 增加 YAML 配置化，便于批量实验。

### 8.2 环境耦合高

零样本三个模型建议独立环境，操作成本高。

建议：

- 提供一键脚本（如 `scripts/setup_env_*.sh` 或 PowerShell 版）；
- 增加最小可运行 smoke test。

### 8.3 参数敏感性文档可加强

`tau/theta/m/tau_f/alpha/beta` 直接决定伪标注质量。

建议：

- 在文档中给出参数调优建议和典型失败案例；
- 输出更多诊断统计（每类保留率、每图伪框数分布）。

### 8.4 README 适配中文使用者

目前以英文为主，国内用户上手门槛偏高。

建议：

- 增补中文版快速开始文档（数据准备、常见报错、Windows 注意项）。
